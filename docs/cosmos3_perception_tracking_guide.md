# Cosmos 3 世界模型下的环境目标轨迹预测与闭环评估方案指南

本指南详细介绍了在 NVIDIA Cosmos 3 世界模型仿真链路中，如何结合**零样本目标检测模型（LocateAnything）**与**基于规则的物体跟踪算法（如 SORT/ByteTrack）**，建立起对动态环境车辆和障碍物的大地坐标系（WGS-84）轨迹预测与评估系统。

---

## 1. 核心概念与定位

### 1.1 Cosmos 世界模型的作用
* **生成式仿真物理引擎**：Cosmos 3 的前向动力学（Forward Dynamics）模型是一个**像素级生成模型**。在给定初始图像和自车动作（Action）后，它能够预测并渲染出极其逼真的未来驾驶场景视频。
* **内生交通动态**：生成的视频中，其他车辆、行人的避让、跟车、变道等行为均是 Cosmos 内部学习到的物理世界演化逻辑，无需人工为背景车编写繁琐的逻辑规则。

### 1.2 为什么需要外接感知与跟踪链路？
由于 Cosmos 3 的输出是**非结构化的像素视频**，并不包含三维边界框（3D Bounding Boxes）或跟踪 ID 的结构化输出。为了获得环境中其他车辆的结构化大地坐标轨迹，必须通过“**Cosmos 仿真渲染 + 外部感知检测与跟踪**”的闭环链路进行提取。

---

## 2. 闭环仿真的感知与跟踪链路设计

整个闭环提取流程分为四个核心阶段：

```
+------------------+     +-----------------------+     +-------------------+
|  Cosmos 世界模型  | --> |  LocateAnything (2D)  | --> |  SORT / ByteTrack |
|  生成仿真视频帧  |     |  零样本目标检测与分割 |     |  基于规则的多目标跟踪|
+------------------+     +-----------------------+     +-------------------+
                                                                 |
                                                                 v
+------------------+     +-----------------------+     +-------------------+
|  大地坐标输出     | <-- |   逆透视投影 (IPM)     | <-- |  相机相对 3D 坐标 |
|  (WGS-84 JSON)   |     |   或单目深度模型估算   |     |  (X_cam, Y_cam, Z_cam)
+------------------+     +-----------------------+     +-------------------+
```

### 2.1 检测阶段：LocateAnything (2D)
在 Cosmos 预测生成的每一帧图像中，利用通用零样本检测模型（如 Grounding DINO 或 Segment Anything）对动态目标进行识别：
* **Prompt 词输入**：使用 `"vehicle"`, `"car"`, `"pedestrian"` 等文本提示词。
* **输出**：每帧中所有候选目标的 2D 边界框 $B_i = [x_{min}, y_{min}, x_{max}, y_{max}]$。

### 2.2 跟踪阶段：基于规则的跟踪器 (SORT / ByteTrack)
通过轻量级、无需训练的经典追踪算法将检测框跨帧关联，维持目标的 ID 一致性：
* **卡尔曼滤波（Kalman Filter）**：预测上一帧各目标在当前帧的位置。
* **匈牙利算法匹配（Hungarian Algorithm）**：计算预测位置与当前检测框的交并比（IoU），实现最优匹配。
* **输出**：生成连续的带有唯一 ID 的车辆轨迹（如 `car_101` 在 1~100 帧的 2D 像素位置）。

### 2.3 三维测距与坐标估算（2D 转 3D）
要将图像上的 2D 目标框转化为相机坐标系下的 3D 物理坐标 $(X_{cam}, Y_{cam}, Z_{cam})$，可采用以下两种方案之一：

#### 方案 A：逆透视投影（IPM，基于地面假设）—— 完全基于规则
* **适用条件**：平坦路面，且已知摄像头的安装高度 $H_{cam}$、下俯角 $\theta_{pitch}$ 以及相机内参矩阵 $K$。
* **方法**：取 2D 边界框的**底边中点**（轮胎接地点 $P_{pixel} = [u_{bottom}, v_{bottom}]$），利用单目相机成像的几何投影公式逆向求解，将该点映射到三维地面。
* **公式简述**：
  $$Z_{cam} = \frac{f \cdot H_{cam}}{v_{bottom} - v_{principal\_point}}$$
  由此可直接算出车辆距离相机的物理深度 $Z$（前后）以及水平偏移 $X$（左右）。

#### 方案 B：单目深度估计模型（如 Depth Anything V2）
* **方法**：对生成的帧运行深度估计网络，提取检测框内部像素的深度值中位数，直接获得物理距离 $Z_{cam}$。

## 3. 坐标转换与大地坐标系输出

一旦通过感知阶段获得了自车和环境中其他车辆在每一帧相对于相机的 3D 坐标，即可通过 [run_action_custom.py](file:///home/qingxu/software/cosmos-Cosmos3/cookbooks/cosmos3/generator/action/run_action_custom.py) 脚本中的几何模块进行转换。

### 3.1 相机坐标系向局部 ENU 坐标系的映射
Cosmos 使用 OpenCV 的相机坐标约定（+$X$ 为右，+$Y$ 为下，+$Z$ 为前向）。局部大地（ENU，东-北-天）坐标系与其转换关系如下：
* **东方向（East）**： $X_{enu} = X_{cam}$
* **北方向（North）**：$Y_{enu} = Z_{cam}$
* **天方向（Up）**：   $Z_{enu} = -Y_{cam}$

对于自车，先通过 `pose_rel_to_abs` 还原绝对相机姿态矩阵，提取出各帧的平移向量，即可得到自车局部的 ENU 坐标。

### 3.2 大地坐标（WGS-84）的转换
利用高斯-克吕格投影的局部平面简化模型，以初始位置或指定参考点 $(Ref_{lat}, Ref_{lon}, Ref_{alt})$ 为基准，将局部的米级偏移量 $(X_{enu}, Y_{enu}, Z_{enu})$ 转化为经纬度：

$$Latitude = Ref_{lat} + \frac{Y_{enu}}{M_{lat}}$$
$$Longitude = Ref_{lon} + \frac{X_{enu}}{M_{lon}}$$
$$Altitude = Ref_{alt} + Z_{enu}$$

其中 $M_{lat}$ 与 $M_{lon}$ 分别为参考点纬度处，每度纬度与每度经度对应的地球弧长（米）。

## 4. 接口集成规范

[run_action_custom.py](file:///home/qingxu/software/cosmos-Cosmos3/cookbooks/cosmos3/generator/action/run_action_custom.py) 支持读取并解析包含环境目标轨迹的 JSON 输入，格式定义如下：

```json
{
  "reference_point": {
    "latitude": 31.2304,
    "longitude": 121.4737,
    "altitude": 0.0
  },
  "ego_trajectory": [
    { "timestamp": 0.0, "x": 0.0, "y": 0.0, "z": 0.0, "roll": 0.0, "pitch": 0.0, "yaw": 0.0 },
    { "timestamp": 0.1, "x": 0.0, "y": 1.35, "z": 0.0, "roll": 0.0, "pitch": 0.0, "yaw": 0.0 }
  ],
  "environmental_vehicles": {
    "car_101": [
      { "timestamp": 0.0, "x": 3.5, "y": 5.0, "z": 0.0 },
      { "timestamp": 0.1, "x": 3.5, "y": 6.5, "z": 0.0 }
    ]
  },
  "environmental_obstacles": {
    "light_pole_0": { "x": 2.0, "y": 20.0, "z": 0.0 }
  }
}
```

运行自定义测试脚本时，程序会自动计算生成符合时长规范的模拟视频，并将转换好的自车及所有动态背景车、障碍物在**大地坐标系下（WGS-84）**的经纬度数据统一输出至目标 JSON。
